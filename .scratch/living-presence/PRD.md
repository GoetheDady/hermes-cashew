# PRD: Living Presence

Status: 待分诊

## Problem Statement

Subtle Motion 让 Hermes Desktop 的状态变化有了呼吸感——消息入场、输入框聚焦、thinking block 展开等过渡不再生硬。但用户在这些交互之间、或者还没开始打字时，面对的是一个静态的空态画面：一个呼吸圆点 + 一句固定文案。盯 10 秒之后，呼吸的规律被看穿，它就变成了"死的动画"。

用户希望产品不只是"过渡时有动画"，而是在**空闲时也感觉 Hermes 在场**——像一个真的在等待的助手，而不是一个等待指令的工具。这种感觉不来自拟人化角色或装饰性背景，而来自**有时间感的、动态变化的、由 Hermes 自己产生的内容**。

## Solution

在单一对话页面的空态（空闲时）注入三层生命力：

1. **时间感知**：空态问候语随系统时间变化（早上好 / 下午好 / 晚上好 / 夜深了），让用户感觉 Hermes 知道现在几点。
2. **Hermes 自生成空闲文案**：不再写死轮播文案。后端维护一个 Presence Engine，通过独立的 WebSocket 连接到 Hermes dashboard，周期性让 Hermes 自己生成简短、温暖、口语化的中文问候语。前端从 REST 端点拉取并慢速轮播展示。
3. **有机呼吸**：将现有的固定周期脉冲升级为双频叠加的不规则呼吸——两个不同频率的正弦波叠加，产生永不精确重复的节律，类似生物的自然呼吸。

产品原则沿用 Subtle Motion 的方向：

- "界面应有呼吸感"——但现在呼吸是活的，不是机械的。
- Motion 仍是细笔，不是烟花——加入的所有变化都应该安静、克制。
- 不拟人化、不加角色、不加装饰背景——生命力来自 Hermes 自己说的话，不是视觉特效。

## User Stories

1. As a Hermes Desktop user, I want the empty conversation state to show a time-aware greeting, so that Hermes feels aware of the time of day.
2. As a Hermes Desktop user, I want the empty state greeting to change across morning / afternoon / evening / night, so that the assistant feels present in real time.
3. As a Hermes Desktop user, I want the idle prompt text to change slowly while I am not typing, so that the screen never feels frozen.
4. As a Hermes Desktop user, I want the idle prompts to feel natural and varied, so that I do not notice a mechanical rotation pattern.
5. As a Hermes Desktop user, I want the idle prompts to be generated by Hermes itself, so that they feel like the assistant's own thoughts rather than pre-written marketing copy.
6. As a Hermes Desktop user, I want the breathing indicator to feel organic rather than mechanical, so that the assistant's presence feels alive even in stillness.
7. As a Hermes Desktop user, I want the pulsing glow on the empty state to have an irregular, natural rhythm, so that it does not become predictable after a few seconds.
8. As a Hermes Desktop user, I want all idle-state motion to remain subtle, so that the app keeps its quiet working-tool personality.
9. As a Hermes Desktop user, I do not want background animations, decorative particles, or gradient orbs, so that the focus remains on the conversation surface.
10. As a Hermes Desktop user, I do not want avatars, mascots, or personality layers, so that Hermes does not become over-personified.
11. As a Hermes Desktop user, I want the idle state changes to respect reduced-motion preferences, so that motion-sensitive users are not disturbed.
12. As a Hermes Desktop user, I want the time-aware greeting and idle prompts to disappear as soon as I start typing, so that the UI never competes with my input.
13. As a Hermes Desktop user, I want the connection state indicators to remain present in the idle state, so that I can still see whether Hermes is ready when I am not typing.
14. As a Hermes Desktop maintainer, I want the Presence Engine to run in the Node backend, so that prompt generation does not depend on the renderer being visible.
15. As a Hermes Desktop maintainer, I want the Presence Engine to use a dedicated Hermes session, so that idle prompt generation never interferes with user conversations.
16. As a Hermes Desktop maintainer, I want the Presence Engine to expose a simple REST endpoint, so that the frontend can poll for prompts without managing a WebSocket connection.
17. As a Hermes Desktop maintainer, I want the Presence Engine to handle generation failures gracefully, so that the app always has fallback prompts to display.
18. As a Hermes Desktop maintainer, I want the Presence Engine to clean up its WebSocket connection on shutdown, so that no orphan connections remain.
19. As a Hermes Desktop maintainer, I want the frontend idle prompt hook to poll at a reasonable interval, so that new prompts appear without excessive network traffic.
20. As a Hermes Desktop maintainer, I want the frontend idle prompt rotation to use irregular intervals, so that the text changes feel organic rather than scheduled.
21. As a Hermes Desktop maintainer, I want the implementation to keep the existing conversation page structure, so that the product surface remains unchanged.
22. As a Hermes Desktop maintainer, I want the dashboard token to remain in the backend, so that the security boundary is preserved.
23. As a Hermes Desktop maintainer, I want the implementation to follow existing code patterns, so that the codebase remains consistent.

## Implementation Decisions

### Architecture

- Add a `PresenceEngine` module in the Node backend. It opens an independent WebSocket connection to the Hermes dashboard (using the `ws` library, same pattern as `proxy.ts`), creates a dedicated session via `session.create`, and periodically sends `prompt.submit` with a lightweight prompt asking Hermes to generate idle greeting text.
- The Presence Engine maintains a ring buffer of generated prompts (maximum 10 items). On startup it generates the first batch immediately; thereafter it generates a new batch every 5 minutes.
- The generation prompt is crafted to produce short, warm, natural Chinese greetings: "生成5条简短的问候语，每条不超过20个汉字。你就是这个对话助手的内心独白，像一个朋友在旁边等待时随口的轻声话语。语气温暖自然，口语化，不要书面语。输出格式：每行一条，不要编号，不要引号，不要任何额外说明。"
- Responses are parsed by splitting on newlines, filtering empty lines, stripping quotes, and truncating overly long lines.
- Expose a `GET /api/presence/prompts` REST endpoint that returns the current ring buffer as a JSON array. This follows the same routing pattern used by `GET /api/sessions` (URL matching in `index.ts`, separate handler module in `routes/`).
- If the buffer is empty (startup not yet complete, or all generations failed), the endpoint returns a built-in fallback list so the frontend always has something to display.

### Frontend

- Extract `EmptyConversation` from the Chat route into its own component file. The Chat route imports it instead of defining it inline.
- Add a `useIdlePrompts` hook that fetches from `/api/presence/prompts` on mount and every 60 seconds thereafter. It maintains a current index that advances at random intervals between 8 and 15 seconds via recursive `setTimeout`.
- Add a `useTimeGreeting` hook that returns a time-based Chinese greeting ("早上好" / "中午好" / "下午好" / "晚上好" / "夜深了，我还在。") and refreshes every 60 seconds.
- Replace the single breathing dot in the empty state with a dual-layer breathing indicator: two nested motion spans with different `animate` keyframe arrays and different `transition.duration` values (e.g., 1.4s and 2.3s), producing an irregular beat-pattern through frequency superposition.
- The idle prompt rotation uses `AnimatePresence` with a slow fade transition (0.8s ease-in-out, y: ±4px) so the text change feels like a gentle shift, not a pop.
- When the user starts typing their first message, the empty state naturally scrolls away — no exit animation needed.
- All motion checks `useReducedMotion()` and falls back to static display when the preference is set.

### Organic Breathing Design

- Layer 1 (inner dot): `opacity: [0.35, 0.9, 0.35]`, `scale: [1, 1.18, 1]`, 1.4s period
- Layer 2 (outer glow ring): `opacity: [0.55, 0.95, 0.55]`, `boxShadow: [0_0_0_3px_...primary_10%, 0_0_0_7px_...primary_16%, 0_0_0_3px_...primary_10%]`, 2.3s period
- The frequency ratio 1.4:2.3 is not a simple rational fraction, so the combined visual pattern repeats only after ~32 seconds — well beyond what a user would track consciously.
- When connecting or session-starting, use a muted version (muted-foreground instead of primary color) with the same dual-layer approach.

### Domain Language

- Use "Presence Engine" for the backend module that generates idle prompts.
- Use "idle prompt" for the rotating text lines in the empty state.
- Use "time-aware greeting" for the hour-based salutation.
- Use "organic breathing" for the dual-frequency pulse.
- Keep "Gateway Reconnect" semantics unchanged.
- Keep "Session History" semantics unchanged — it is not visible in the default product surface.
- The dashboard token remains exclusively in the backend.

## Testing Decisions

- The primary testing seam is the conversation page's empty state behavior.
- A good test verifies that the empty state renders with a time-appropriate greeting when the gateway is ready and no session is active.
- A good test verifies that idle prompts rotate over time without blocking the input box.
- A good test verifies that the empty state disappears when messages are added to the conversation (user starts typing).
- A good test verifies that the Presence Engine creates a WebSocket connection, sends a session.create request, and eventually populates the prompt buffer.
- A good test verifies that the REST endpoint returns the current buffer (or fallback prompts if empty).
- A good test verifies that the Presence Engine stops cleanly without leaving orphan connections.
- Tests should not assert animation timing values, DOM animation wrappers, or specific prompt text content.
- Existing typecheck, lint, and Electron/Vite build must continue to pass.
- Manual QA should inspect the empty state over a 2-minute period to confirm: greeting changes with time, idle prompts rotate at irregular intervals, breathing pattern does not feel mechanically repetitive, and starting to type immediately transitions to the conversation view.
- Manual QA should verify that changing the system clock across hour boundaries updates the greeting within 60 seconds.
- Manual QA should verify that reduced-motion mode disables all breathing and rotation animations while keeping the text content present.

## Out of Scope

- Adding personality layers, character design, or mascots.
- Adding background animations, particles, gradient orbs, or decorative visuals.
- Adding sound effects.
- Adding page transitions or route-level animation.
- Changing the conversation page layout or message bubble design.
- Changing Gateway Reconnect, session creation, or prompt submission behavior.
- Changing the dashboard token ownership model.
- Adding a full test framework for animation timing assertions.
- Making the idle prompts interactive or clickable.
- Displaying idle prompts when messages are present in the conversation.
- Persisting generated prompts across backend restarts.
- Adding user configuration for idle prompt frequency or content.
- Exposing the Presence Engine session in Session History.

## Further Notes

- This PRD extends the Subtle Motion philosophy from "the interface has breath" to "Hermes feels present." The leap is from passive animation to active, dynamic content.
- The Presence Engine is intentionally simple — a ring buffer, not a database. If the backend restarts, prompts are regenerated fresh. This is acceptable for an MVP where prompt generation is lightweight.
- The 5-minute regeneration interval and 10-item buffer size are starting points. They can be tuned based on observed token usage and prompt freshness.
- The idle prompt generation prompt itself may evolve over time based on the quality and variety of outputs observed in practice.
- The organic breathing dual-frequency approach is a pragmatic approximation of natural variation, not a physiologically accurate model. It is chosen because it produces the desired perceptual effect with minimal code complexity.
